会话管理：按任务阶段拆，不按角色机械拆

会话推进阶段，文件完成交接。

协作方式

探索会话

适用于想法研讨、需求澄清、原型确认、技术路线比较。

  • 不急着写代码
  • 输出 PROJECT_DOC 草案或 change 草案
  • 确认做什么、不做什么、验收标准

执行会话

适用于已经明确目标的开发、修复、测试、重构。

  • 按 PROJECT_DOC 或 changes 开发
  • 完成后更新 CURRENT_STATUS
  • 验证通过后按规则 commit

运维 / 复盘会话

适用于线上异常、用户反馈、运营数据、版本复盘。

  • 先事实，后判断
  • 区分已确认事实和推测
  • 把反馈抽象成 change request

0-1 与变更请求的不同处理

0-1 建立基线；有基线后，需求走增量。

决策规则

写 PROJECT_DOC直接把确认内容进入产品和技术基线。
不建 changes因为还没有稳定基线，不需要增量文件。

开发后更新状态完成后同步 CURRENT_STATUS。

日常操作检查清单

让 Codex 每次开工和收尾都有清晰边界。

✓

✓

✓
