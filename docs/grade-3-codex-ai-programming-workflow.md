# 三年级：Codex AI 编程工作流总览

这份资源帮助你把“用 AI 写代码”升级成“用 AI 管理一个项目的研发上下文”。它适合放在三年级 AI 编程部分，作为 Easy-Vibe、Vibe Coding 和工程化工具课程之后的工作流补充。

## 适合谁

- 已经会让 AI 生成代码，但项目一复杂就容易忘记前因后果的人。
- 经常在不同会话里反复解释项目背景、当前状态和开发规则的人。
- 想让 Codex 更稳定地参与需求澄清、实现、验证、部署和复盘的人。

## 解决什么问题

AI 编程真正困难的地方，不只是“让模型写出代码”，而是让它在每次工作时拿到最小、最新、最可信的上下文。

如果所有信息都散落在聊天记录里，AI 很容易遇到这些问题：

- 新会话不知道项目真实进度。
- 旧需求、当前状态、下一步混在一起。
- 开发规则靠口头重复，执行不稳定。
- 需求变更没有沉淀，后续很难追踪。
- 验证、部署、复盘没有闭环记录。

这份工作流的核心思路是：**长期事实放进文件，当前阶段交给会话推进。**

## 核心原则

1. 0-1 阶段先建立基线，不急着拆变更单。
2. 已有产品新增需求，先进入 `docs/changes/*.md`。
3. 每次开发、修复、部署或重要调研后，更新 `CURRENT_STATUS.md`。
4. 会话不保存长期事实，文件保存长期事实。

## 研发流程

一个稳定的 AI 编程项目，可以按下面闭环推进：

1. **想法研讨**：澄清用户、场景、问题和目标。
2. **图片 / 原型**：确认产品形态和关键交互。
3. **PROJECT_DOC.md**：建立当前产品与技术基线。
4. **Codex 执行**：按基线或变更单落地。
5. **验证与交付**：完成测试、手工验收和风险说明。
6. **CURRENT_STATUS.md**：记录当前事实、验证结果、未验证风险和下一步。
7. **changes/*.md**：把反馈和新需求沉淀为下一轮变更。

## 两级 AGENTS.md

### 全局 AGENTS.md

位置示例：`C:\Users\yuhui\.codex\AGENTS.md`

它适合放通用工作方法：

- 硬性禁止和默认工作方式。
- 项目上下文读取顺序。
- 新项目初始化规则。
- 通用交付与缺陷解释标准。

### 项目 AGENTS.md

位置示例：`your-project/AGENTS.md`

它适合放项目专属细节：

- 本项目上下文入口。
- 启动、构建、测试、Lint 命令。
- 项目目录和开发约定。
- 本项目验证要求和禁止事项。

不要把模板正文和项目事实都塞进全局 AGENTS.md。全局管方法，项目管细节。

## 上下文文件矩阵

| 文件 | 它回答什么 | 什么时候维护 |
| --- | --- | --- |
| `PROJECT_CONTEXT.md` | 这个产品长期是什么？哪些事实稳定不变？ | 低频维护；业务背景或边界变化时更新 |
| `PROJECT_DOC.md` | 当前产品做什么？怎么做？如何验收？ | 0-1 阶段建立；基线变化时更新 |
| `CURRENT_STATUS.md` | 现在做到哪？还差什么？下一步是什么？ | 高频维护；每次实现、修复、部署、重要调研后更新 |
| `CHANGELOG.md` | 过去每个版本对用户改了什么？ | 上线或用户可见版本变化后更新 |
| `docs/changes/*.md` | 相对当前产品，这次为什么改、改什么、不改什么？ | 已有产品新增需求前创建，开发后归档为事实 |

## 会话管理

不要按角色机械拆会话，而是按任务阶段拆会话。

### 探索会话

适用于想法研讨、需求澄清、原型确认、技术路线比较。

输出通常是 `PROJECT_DOC.md` 草案或 change 草案。

### 执行会话

适用于已经明确目标的开发、修复、测试、重构。

执行会话应按 `PROJECT_DOC.md` 或 `docs/changes/*.md` 开发，完成后更新 `CURRENT_STATUS.md`，验证通过后提交。

### 运维 / 复盘会话

适用于线上异常、用户反馈、运营数据、版本复盘。

先确认事实，再做判断。推测必须标注，并把可复用反馈抽象成 change request。

## 0-1 与已有产品的不同处理

### 0-1 阶段

- 先研讨和具象化。
- 再写 `PROJECT_DOC.md`。
- 不急着建 `docs/changes/*.md`，因为还没有稳定基线。

### 已有产品

- 新需求先写 change。
- 开发后更新 `CURRENT_STATUS.md`。
- 只有影响产品或技术定义时，才回写 `PROJECT_DOC.md`。

## 日常操作检查清单

- 开工先读 `CURRENT_STATUS.md`。
- 只有涉及产品或技术基线时，再读取 `PROJECT_CONTEXT.md` / `PROJECT_DOC.md`。
- 新增需求先建 `docs/changes/*.md`。
- 实现后更新 `CURRENT_STATUS.md`。
- 用户可见发布更新 `CHANGELOG.md`。
- 会话交接靠文件，不靠旧聊天记录。
- 子智能体适合并行读、查、审、测，避免同时改同一批核心文件。

## 结业任务

请用自己的项目完成一次工作流整理：

1. 写出项目当前最重要的 3 个长期事实。
2. 写出当前真实状态、已验证内容、未验证风险和下一步。
3. 判断当前需求属于 0-1 基线，还是已有产品增量。
4. 为下一次 Codex 会话准备一段 150-300 字的任务说明。
5. 完成一次小改动后，更新 `CURRENT_STATUS.md` 并写出验证方式。

完成后，你应该能明显减少“每次都要重新解释项目”的成本，让 AI 编程从一次性对话变成可持续协作。
