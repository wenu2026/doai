# 三年级：从想法到可演示应用

适合谁：已经能用 AI 完成日常任务，想把一个想法做成可演示作品的人。

你会得到什么：一个从想法、需求、页面、数据、功能到演示的 AI 项目实践流程。你不需要一开始就成为程序员，但需要学会把目标说清楚、把结果验收清楚。

建议周期：3-7 天完成一个最小版本。

## 项目原则

- 先做可演示版本，不做完整产品。
- 先做一个核心流程，不做所有功能。
- 先用假数据验证体验，再考虑真实数据库。
- 先人工检查结果，再考虑自动化。
- 每一步都让 AI 产出可检查的东西。

## 第一步：把想法写成一句话

模板：

```text
我想做一个{工具/页面/应用}，
帮助{目标用户}
在{具体场景}
完成{具体任务}，
最终得到{可见结果}。
```

示例：

```text
我想做一个客户跟进摘要工具，
帮助销售主管
在每天复盘销售记录时
快速看懂客户关注点和下一步动作，
最终得到一份统一格式的跟进摘要。
```

检查：

- 是否有明确用户？
- 是否有明确场景？
- 是否有明确任务？
- 是否有可见结果？

## 第二步：缩小到一个核心流程

不要把第一版做成“大平台”。先写一个主流程。

```text
用户打开页面后：
1. 
2. 
3. 
4. 
5. 

用户最终看到：
```

示例：

```text
用户打开页面后：
1. 粘贴客户沟通记录。
2. 选择输出类型：摘要 / 跟进建议 / 风险提醒。
3. 点击生成。
4. 页面展示结构化结果。
5. 用户复制结果到工作文档。

用户最终看到：客户关注点、成交阻力、下一步动作、待确认信息。
```

## 第三步：写需求卡片

每个功能写成一张需求卡。

```text
需求名称：

用户是谁：

用户要做什么：

为什么需要：

输入是什么：

输出是什么：

验收标准：

不做什么：
```

示例：

```text
需求名称：生成客户跟进摘要

用户是谁：销售主管

用户要做什么：把销售沟通记录变成统一格式摘要

为什么需要：减少阅读时间，快速判断下一步动作

输入是什么：一段客户沟通记录

输出是什么：客户关注点、客户顾虑、建议动作、待确认信息

验收标准：
- 输出必须分为 4 个栏目。
- 没有输入的信息不能编造。
- 待确认信息要单独列出。

不做什么：
- 不判断客户归属。
- 不自动发送给客户。
- 不做 CRM 系统集成。
```

## 第四步：让 AI 生成页面草图

提示词：

```text
请根据下面需求，帮我设计一个单页应用的页面结构。

需求：
{粘贴需求卡片}

请输出：
1. 页面包含哪些区域
2. 每个区域有什么控件
3. 用户操作流程
4. 空状态、生成中、成功、失败状态
5. 移动端需要注意什么

先不要写代码，只输出产品结构。
```

验收标准：

- 页面能支持核心流程。
- 不出现第一版不需要的功能。
- 有输入、操作、输出和错误提示。

## 第五步：准备示例数据

没有示例数据，就无法判断应用是否可用。

```text
示例输入 1：

期望输出 1：

示例输入 2：

期望输出 2：

异常输入：

期望错误提示：
```

示例：

```text
异常输入：今天聊了下，客户还行。

期望错误提示：当前记录信息不足，请补充客户需求、顾虑、预算、下一步时间。
```

## 第六步：让 AI 辅助实现

如果你使用 AI 编程工具，可以这样开始：

```text
我要做一个最小可演示版本。

项目目标：
{一句话项目}

核心流程：
{粘贴核心流程}

页面结构：
{粘贴页面结构}

示例数据：
{粘贴示例数据}

请先帮我完成：
1. 文件结构建议
2. 页面组件拆分
3. 第一版实现步骤
4. 本地验证方式

不要一次性扩展功能。
```

继续推进时，每次只让 AI 做一个小任务：

- 先做静态页面。
- 再做输入和输出状态。
- 再接入生成逻辑。
- 再处理错误状态。
- 最后优化文案和样式。

## 第七步：检查项目质量

最小可演示版本不等于粗糙版本。交付前检查：

```text
用户体验：
- 第一次打开是否知道该做什么？
- 输入框和按钮是否清楚？
- 生成中是否有反馈？
- 失败时是否知道怎么修正？

功能：
- 正常输入是否能得到结果？
- 空输入是否有提示？
- 异常输入是否能处理？
- 输出是否符合格式？

内容：
- 文案是否是用户视角？
- 是否有内部开发说明？
- 是否有不该对外展示的调试信息？

风险：
- 是否提示用户核对结果？
- 是否避免上传敏感信息？
- 是否避免 AI 编造确定结论？
```

## 第八步：准备演示脚本

演示不是介绍功能列表，而是讲一个用户完成任务的过程。

```text
演示脚本

1. 用户是谁：
2. 他原来遇到什么问题：
3. 现在打开这个工具：
4. 他输入什么：
5. 工具输出什么：
6. 他如何使用结果：
7. 这个版本还不做什么：
8. 下一步最值得改进的是：
```

## 三年级结业任务

完成一个可演示应用或工具原型，并提交：

```text
1. 一句话项目说明：
2. 目标用户：
3. 核心流程：
4. 需求卡片：
5. 页面截图或演示地址：
6. 示例输入和输出：
7. 已验证功能：
8. 未实现功能：
9. 风险和人工复核点：
10. 下一版计划：
```

## 通过标准

达到下面 5 条，就可以进入四年级：

- 有一个真实用户和真实场景。
- 有一个能跑通的核心流程。
- 有示例输入和期望输出。
- 有用户能看懂的演示。
- 能说清当前版本的边界和风险。
